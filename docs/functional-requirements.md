## 1. 機能要件の詳細化

### A. フォルダ管理

#### A-1. フォルダの登録・解除

**ユーザーストーリー**
> ユーザーとして、任意のフォルダをアプリに登録して、その中のMarkdownファイルをサイドバーで一覧・管理したい。不要になったフォルダは解除して表示から除外したい。

**受入基準**
- [ ] Finderからフォルダをドラッグ&ドロップしてサイドバーに登録できる
- [ ] メニューバーまたはツールバーの「フォルダを追加」ボタンからNSOpenPanelでフォルダを選択・登録できる
- [ ] 登録されたフォルダはアプリ再起動後も保持される（UserDefaults or Bookmarkデータで永続化）
- [ ] サンドボックス環境でのセキュリティスコープ付きブックマークを使用し、再起動後もアクセス権を維持する
- [ ] フォルダを右クリック→「登録解除」で一覧から除外できる
- [ ] 登録解除時に確認ダイアログを表示する（ファイル自体は削除しない旨を明示）
- [ ] 登録解除してもディスク上のファイルには一切影響しない

#### A-2. 複数ルートフォルダの同時管理

**ユーザーストーリー**
> ユーザーとして、仕事用・個人用・プロジェクト別など、複数のルートフォルダを同時にサイドバーに表示して切り替えなしに横断的にファイルにアクセスしたい。

**受入基準**
- [ ] サイドバーに複数のルートフォルダをツリー形式で同時表示できる
- [ ] 各ルートフォルダは折りたたみ/展開が可能
- [ ] ルートフォルダの表示順序をドラッグ&ドロップで変更できる
- [ ] 登録上限は設けない（ただしパフォーマンス目標として10フォルダ以上を想定）
- [ ] 各ルートフォルダ名の横にファイル数バッジを表示する（任意設定）

#### A-3. ファイル変更の自動検知と反映

**ユーザーストーリー**
> ユーザーとして、Finder上やVSCodeなど外部エディタでファイルを追加・削除・リネームした際に、アプリのサイドバーが自動的に最新状態に更新されてほしい。

**受入基準**
- [ ] `FSEvents`（macOS FileSystem Events API）を使用してファイルシステムの変更をリアルタイム監視する
- [ ] ファイルの追加・削除・リネーム・移動がサイドバーに1秒以内に反映される
- [ ] ファイル内容の外部変更時、プレビュー中であれば自動で再レンダリングする
- [ ] 編集中のファイルが外部変更された場合はコンフリクト処理を行う（後述 3章参照）
- [ ] 監視のCPU負荷は通常時0.5%以下に抑える

#### A-4. .mdファイルのフィルタリング

**ユーザーストーリー**
> ユーザーとして、サイドバーにはMarkdownファイルとフォルダ構造のみが表示され、画像やPDFなど無関係なファイルでツリーが煩雑にならないようにしたい。

**受入基準**
- [ ] デフォルトでは `.md` および `.markdown` 拡張子のファイルのみ表示する
- [ ] フォルダは中に対象ファイルが1つ以上ある場合のみ表示する（空フォルダ非表示）
- [ ] 設定画面で対象拡張子を追加可能（例: `.txt`, `.mdx`）
- [ ] `.` で始まる隠しフォルダ（`.git`, `.obsidian`など）はデフォルト非表示
- [ ] `node_modules` などの除外パターンを設定可能
- [ ] フィルタのON/OFFをトグルで切り替え可能（全ファイル表示モード）

### B. ファイルプレビュー

#### B-1. Markdown→HTMLレンダリング

**ユーザーストーリー**
> ユーザーとして、サイドバーでファイルを選択したら即座にレンダリング済みのプレビューが表示され、Markdownの構造を視覚的に把握したい。

**受入基準**
- [ ] CommonMark仕様に完全準拠したレンダリングを行う
- [ ] GFM（GitHub Flavored Markdown）拡張にも対応する
- [ ] ファイル選択からプレビュー表示まで200ms以内（通常サイズのファイル）
- [ ] プレビューはWKWebViewベースで表示する
- [ ] CSSテーマを選択可能（ライト/ダーク、最低2種）
- [ ] macOSのダークモード切替に自動追従する

#### B-2. 対応Markdown要素一覧

**ユーザーストーリー**
> ユーザーとして、一般的なMarkdown記法がすべて正しくレンダリングされ、技術ドキュメントやノートを不自由なく閲覧したい。

**受入基準 - 必須要素（MVP）**
- [ ] 見出し（h1〜h6）
- [ ] 段落・改行
- [ ] 強調（太字・斜体・取り消し線）
- [ ] リスト（順序付き・順序なし・ネスト）
- [ ] リンク（インライン・参照形式）
- [ ] 画像（インライン・参照形式）
- [ ] コードブロック（シンタックスハイライト付き）
- [ ] インラインコード
- [ ] 引用（blockquote）
- [ ] 水平線
- [ ] テーブル（GFM形式）
- [ ] チェックリスト（タスクリスト）

**受入基準 - 拡張要素（v1.0）**
- [ ] 数式レンダリング（KaTeX、インライン `$...$` およびブロック `$$...$$`）
- [ ] Mermaidダイアグラム（```mermaid コードブロック）
- [ ] 脚注
- [ ] 定義リスト
- [ ] 目次の自動生成（TOC）

**受入基準 - 高度な要素（v2.0）**
- [ ] YAML Front Matterの解析と表示
- [ ] カスタムコンテナ（admonition / callout）
- [ ] 埋め込みコンテンツ（YouTube等のoEmbed）
- [ ] PlantUMLダイアグラム

#### B-3. 画像の表示

**ユーザーストーリー**
> ユーザーとして、Markdownファイル内で参照している画像が、ローカルの相対パス・絶対パスどちらでも正しく表示されてほしい。

**受入基準**
- [ ] Markdownファイルからの相対パスで指定された画像を正しく表示する
- [ ] 絶対パス（`/Users/...`）で指定された画像を表示する
- [ ] `https://` で始まるリモート画像を表示する（ネットワーク接続時）
- [ ] 対応形式: PNG, JPEG, GIF, SVG, WebP
- [ ] 画像が見つからない場合は代替テキスト（alt）とプレースホルダーを表示する
- [ ] 画像のサイズがウィンドウ幅を超える場合は自動縮小する（レスポンシブ）

#### B-4. リンクの処理

**ユーザーストーリー**
> ユーザーとして、Markdown内のリンクをクリックしたとき、内部リンク（他の.mdファイル）はアプリ内で開き、外部リンクはデフォルトブラウザで開いてほしい。

**受入基準**
- [ ] `.md` ファイルへの相対リンクはアプリ内でそのファイルを開く（プレビュー遷移）
- [ ] `http://` / `https://` の外部リンクはデフォルトブラウザで開く
- [ ] アンカーリンク（`#heading`）はプレビュー内でスクロールする
- [ ] リンク先が存在しない場合はツールチップで「リンク先が見つかりません」と警告する
- [ ] Cmd+クリックで新規タブ/分割ビューで開く（v1.0以降）

### C. ファイル編集

#### C-1. テキスト編集の基本機能

**ユーザーストーリー**
> ユーザーとして、プレビューで気になった箇所をその場で素早く修正したい。本格的なエディタではなく、軽微な修正に適した編集機能がほしい。

**受入基準**
- [ ] プレビューと編集のモード切替が可能（トグルボタンまたはショートカット `Cmd+E`）
- [ ] 編集モードではプレーンテキストエディタを表示する
- [ ] 基本的なテキスト操作: 選択、コピー、ペースト、カット、全選択
- [ ] 行番号の表示（ON/OFF切替可能）
- [ ] 基本的なMarkdown構文ハイライト（見出し・太字・リスト等を視覚的に区別）
- [ ] タブキーでスペース挿入（2スペース or 4スペース、設定可能）
- [ ] 編集とプレビューの分割表示（サイドバイサイド）- v1.0以降

#### C-2. 自動保存 vs 手動保存

**ユーザーストーリー**
> ユーザーとして、編集内容が意図せず失われることなく、かつ意図しない変更が保存されることもない、安心できる保存方式がほしい。

**設計方針: ハイブリッド方式（推奨）**

**受入基準**
- [ ] 自動保存をデフォルトで有効にする（入力停止後2秒で自動保存）
- [ ] 自動保存のON/OFFを設定画面で切り替え可能
- [ ] 自動保存OFF時は `Cmd+S` で手動保存
- [ ] 未保存の変更がある場合、タブ/サイドバーにインジケータ（ドット等）を表示する
- [ ] アプリ終了時に未保存の変更がある場合は確認ダイアログを表示する
- [ ] 自動保存時にファイルの最終変更日時を確認し、外部変更とのコンフリクトを検知する

#### C-3. Undo/Redo

**ユーザーストーリー**
> ユーザーとして、編集の取り消し・やり直しを標準的な操作で行いたい。

**受入基準**
- [ ] `Cmd+Z` でUndo、`Cmd+Shift+Z` でRedoが動作する
- [ ] Undoヒストリはファイルごとに独立して管理する
- [ ] Undo回数の上限は100ステップ以上
- [ ] ファイルを閉じるとUndoヒストリはクリアされる
- [ ] 文字単位ではなく意味のある操作単位（単語入力、行削除など）でグルーピングする

#### C-4. 文字エンコーディング

**ユーザーストーリー**
> ユーザーとして、日本語を含むMarkdownファイルが文字化けなく読み書きできてほしい。

**受入基準**
- [ ] UTF-8をデフォルトのエンコーディングとする
- [ ] BOM付きUTF-8も正しく読み込む（BOMは保持して保存）
- [ ] UTF-8以外のエンコーディング（Shift_JIS, EUC-JP等）を検出した場合、ステータスバーに警告を表示する
- [ ] 改行コードはLF/CRLF/CRを自動判定し、元のコードを維持して保存する
- [ ] 新規ファイル作成時はUTF-8（BOMなし）+ LFをデフォルトとする

### D. 検索・フィルタ

#### D-1. ファイル名検索

**ユーザーストーリー**
> ユーザーとして、大量のMarkdownファイルの中から、ファイル名の一部を入力して素早く目的のファイルを見つけたい。

**受入基準**
- [ ] `Cmd+P` でクイックオープン（ファイル名検索）パレットを表示する
- [ ] インクリメンタルサーチで入力に応じてリアルタイムに候補を絞り込む
- [ ] ファジーマッチングに対応（例: "frq" → "functional-requirements.md"）
- [ ] 検索結果にファイルの所属フォルダパスを表示する
- [ ] 矢印キーで候補を選択し、Enterで開く
- [ ] 最近開いたファイルを優先的に上位に表示する

#### D-2. 全文検索（フォルダ横断）

**ユーザーストーリー**
> ユーザーとして、登録した全フォルダのMarkdownファイルを対象に、キーワードで全文検索してヒット箇所を一覧で確認したい。

**受入基準**
- [ ] `Cmd+Shift+F` で全文検索パネルを表示する
- [ ] 登録済み全フォルダを横断して検索する
- [ ] 検索結果をファイルごとにグルーピングして表示する
- [ ] ヒット箇所の前後コンテキスト（前後1行程度）をプレビュー表示する
- [ ] 検索結果をクリックすると該当ファイルの該当箇所にジャンプする
- [ ] 大文字小文字の区別ON/OFFを切り替え可能
- [ ] 正規表現検索に対応（v1.0以降）
- [ ] 検索は非同期で実行し、UIをブロックしない
- [ ] 10,000ファイル規模でも5秒以内に結果を返す

#### D-3. タグ/ラベル機能の要否

**設計判断: MVP/v1.0では見送り、v2.0で検討**

**理由**
- Markdownファイル自体にメタデータを持たせる方式（YAML Front Matter内の `tags:` フィールド）は既存エコシステム（Obsidian, Hugo等）との互換性が高い
- ただしMVPの範囲を超えるため、段階的に導入する
- v2.0でFront Matterのパースを実装した際に、タグの抽出・フィルタリング機能を追加する

**v2.0での受入基準（参考）**
- [ ] YAML Front Matterの `tags:` フィールドを自動認識する
- [ ] サイドバーにタグ一覧パネルを表示する
- [ ] タグをクリックすると該当ファイルをフィルタ表示する
- [ ] タグの一括リネーム・統合機能

## 2. 非機能要件

### 2-1. パフォーマンス目標

| 指標 | 目標値 | 計測条件 |
|------|--------|----------|
| コールドスタート起動時間 | 1.5秒以内 | 5フォルダ・500ファイル登録時 |
| ウォームスタート起動時間 | 0.5秒以内 | 同上 |
| ファイル切り替え→プレビュー表示 | 200ms以内 | 50KB以下の.mdファイル |
| ファイル切り替え→プレビュー表示 | 1秒以内 | 1MB以下の.mdファイル |
| サイドバーのツリー展開 | 100ms以内 | 1,000ファイルのフォルダ |
| 全文検索の応答 | 3秒以内 | 1,000ファイル・平均10KB |
| 全文検索の応答 | 10秒以内 | 10,000ファイル・平均10KB |
| 自動保存の遅延 | 体感なし | ファイルI/Oはバックグラウンドスレッド |

### 2-2. リソース使用量

| 指標 | 目標値 | 条件 |
|------|--------|------|
| アイドル時メモリ | 80MB以下 | 5フォルダ・500ファイル登録 |
| アクティブ時メモリ | 200MB以下 | プレビュー表示中 |
| ピークメモリ | 500MB以下 | 全文検索実行中 |
| アイドル時CPU | 0.5%以下 | ファイル監視のみ |
| ディスク使用量（アプリ本体） | 50MB以下 | - |
| ディスク使用量（キャッシュ等） | 100MB以下 | - |

### 2-3. 対応環境

- macOS 14 (Sonoma) 以降
- Apple Silicon (M1以降) ネイティブ対応
- Intel Mac も Universal Binary でサポート
- 最小画面解像度: 1280x720

### 2-4. アクセシビリティ

- VoiceOverによるサイドバー・プレビューの読み上げ対応
- キーボードのみでの全操作を可能にする
- Dynamic Type（テキストサイズ変更）への対応

## 3. エッジケース・例外処理

### 3-1. 外部でファイルが削除された場合

| 状態 | 振る舞い |
|------|----------|
| サイドバーに表示中（非選択） | サイドバーから自動的に除去する |
| プレビュー表示中 | 「このファイルは削除されました」バナーを表示。プレビュー内容は保持する |
| 編集中（未保存の変更あり） | 警告ダイアログ表示。「名前を付けて保存」または「破棄」を選択させる |
| 編集中（変更なし） | 「このファイルは削除されました」バナーを表示。編集は継続可能 |

### 3-2. 外部でファイルが変更された場合（コンフリクト処理）

| 状態 | 振る舞い |
|------|----------|
| プレビュー表示中（編集なし） | 自動的に最新内容で再レンダリングする |
| 編集中（未保存の変更あり） | コンフリクトダイアログを表示し、以下の選択肢を提示: |
| | - 「外部の変更を取り込む」（ローカル変更を破棄） |
| | - 「自分の変更を維持する」（外部変更を無視して上書き保存） |
| | - 「差分を確認する」（簡易diffビューを表示） |
| 編集中（変更なし） | サイレントに最新内容を反映する |

### 3-3. 権限のないフォルダにアクセスした場合

- フォルダ登録時にアクセス権がない場合: エラーダイアログで「このフォルダへのアクセス権限がありません」と表示
- 登録済みフォルダの権限が後から変更された場合: サイドバーのフォルダアイコンに警告マークを表示し、ツールチップで理由を説明
- サンドボックス環境でのセキュリティスコープブックマークの期限切れ: 再度フォルダ選択を促すダイアログを表示

### 3-4. 壊れたMarkdownファイルの処理

- **方針**: Markdownは「壊れる」ことが基本的にない（プレーンテキストとして必ず表示可能）
- 不正なMarkdown構文: パーサーが解釈できない部分はプレーンテキストとしてそのまま表示する（エラーにしない）
- バイナリファイルが `.md` 拡張子を持つ場合: 「このファイルはテキストファイルとして読み取れません」メッセージを表示
- ヌル文字を含むファイル: ヌル文字を除去して表示し、ステータスバーに警告を表示

### 3-5. シンボリックリンクの扱い

- シンボリックリンクはデフォルトで追従する（リンク先の実体を表示）
- 循環シンボリックリンクの検出: 最大深度（20階層）を設定し、超過した場合はそのブランチの探索を中止
- 壊れたシンボリックリンク（リンク先が存在しない）: サイドバーでグレーアウト表示し、ツールチップで「リンク先が見つかりません」と表示
- 設定でシンボリックリンクの追従ON/OFFを切り替え可能

### 3-6. 非常に大きなMarkdownファイル（10MB+）

| ファイルサイズ | 振る舞い |
|---------------|----------|
| 〜1MB | 通常通り全文レンダリング |
| 1MB〜10MB | 警告バナー「大きなファイルです。表示に時間がかかる場合があります」を表示してからレンダリング |
| 10MB〜50MB | プレビューは先頭部分のみ表示（遅延読み込み / ページネーション）。編集はプレーンテキストモードに制限 |
| 50MB以上 | 「このファイルは大きすぎるため表示できません。外部エディタでの編集を推奨します」メッセージを表示。「外部エディタで開く」ボタンを提供 |

## 4. フェーズ分け

### Phase 0: MVP（最小限で動くプロダクト）

**目標**: 「フォルダを登録して、Markdownを見る」が最速でできること

**期間目安**: 4〜6週間

| カテゴリ | 機能 |
|----------|------|
| フォルダ管理 | 単一フォルダの登録・解除 |
| フォルダ管理 | フォルダツリーの表示（.md/.markdownフィルタ） |
| フォルダ管理 | 隠しフォルダの除外 |
| プレビュー | CommonMark + GFM基本要素のレンダリング |
| プレビュー | シンタックスハイライト付きコードブロック |
| プレビュー | 相対パス画像の表示 |
| プレビュー | ライト/ダークテーマ（macOS追従） |
| リンク | 外部リンクをブラウザで開く |
| UI | サイドバー + プレビューの2ペイン構成 |
| UI | ウィンドウサイズ・位置の保存 |

**含めないもの**: 編集機能、検索機能、複数フォルダ、自動検知

---

### Phase 1: v1.0（実用レベル）

**目標**: 日常的なMarkdown管理ツールとして不自由なく使えること

**期間目安**: MVP後 6〜8週間

| カテゴリ | 機能 |
|----------|------|

## 付録: 技術選定の推奨方針

| 領域 | 推奨技術 | 理由 |
|------|----------|------|
| UI フレームワーク | SwiftUI + AppKit（ハイブリッド） | SwiftUIで全体構成、AppKitで細かいMac体験の調整 |
| Markdownパーサー | swift-markdown (Apple製) or cmark-gfm | 高速・GFM対応・Swift連携が良好 |
| プレビュー表示 | WKWebView | CSS/JSによる柔軟なレンダリング、KaTeX/Mermaidの組み込みが容易 |
| コードハイライト | highlight.js or Prism.js (WebView内) | 言語カバレッジが広い |
| テキスト編集 | NSTextView (TextKit 2) | macOSネイティブの入力体験、IME対応が確実 |
| ファイル監視 | FSEvents API | macOS標準、低負荷、ディレクトリ単位の効率的な監視 |
| 永続化 | UserDefaults + Security-Scoped Bookmarks | サンドボックス対応、軽量 |
| 全文検索 | NSMetadataQuery or カスタムインデックス | Spotlight連携 or 自前のSQLite FTSインデックス |
